# 运维实施手册

> 目标：把部署、监控、备份、回滚、故障处理和数据维护变成固定流程，降低上线和长期维护风险。

## 1. 环境分层

### develop

- 用途：本地开发和微信开发者工具联调。
- 后端 profile：`dev`。
- 微信登录：允许 mock。
- OCR：允许 mock。
- 数据库：本地 MySQL，可随时重置。

### trial

- 用途：小程序体验版、上线前验收。
- 后端 profile：建议使用 `prod`，但连接独立试运行数据库。
- 微信登录：禁止 mock。
- OCR：建议真实调用，至少保留一次真实图片回归。
- 数据库：禁止连接生产库。

### release

- 用途：正式用户。
- 后端 profile：`prod`。
- 微信登录：禁止 mock。
- OCR：真实调用。
- 数据库：生产 MySQL，必须开启备份。

## 2. 发布流程

### 发布前

1. 在本地执行：`bash scripts/verify-project.sh`
2. 确认 GitHub Actions `Verify` 通过。
3. 确认数据库变更：
   - 新表、改表、补列必须有 Flyway migration。
   - `schema.sql` 和 `schema-full-latest.sql` 已同步。
   - SQL 不使用 MySQL 8 专属能力。
4. 确认小程序配置：
   - `trial/release` 使用云托管。
   - `uploadWithSourceMap=false`。
   - 没有超长 `confirmText`。
5. 确认生产环境变量：
   - `SPRING_PROFILES_ACTIVE=prod`
   - `WECHAT_MOCK_ENABLED=false`
   - `AUTH_TOKEN_SECRET` 为 32 位以上高强度随机值
   - 数据库和 OCR 密钥完整

### 发布中

1. 构建镜像：`docker build -t <registry>/xinqi-diet:<tag> .`
2. 推送镜像到腾讯云容器镜像仓库。
3. 云托管发布新镜像。
4. 观察启动日志，确认出现：
   - `activeProfiles=prod`
   - `wechat.mock-enabled=false`
5. 调用 `/readyz`，确认返回：
   - `status=UP`
   - `database=UP`

### 发布后

1. 验证微信登录。
2. 验证首页汇总。
3. 验证饮食记录新增、查询、删除。
4. 验证运动记录新增、查询、删除。
5. 验证体重记录和趋势页。
6. 验证健康日记保存、查询、删除。
7. 抽样数据库：
   - `user_profile.open_id` 不应出现 `mock_openid_`
   - 近 20 条用户记录 `last_login_at` 正常更新

## 3. 监控指标

### 后端健康

- `/api/ping` 存活检查。
- `/readyz` 就绪检查。
- JVM 内存使用率。
- 容器重启次数。
- HTTP 5xx 数量。
- HTTP 401 数量和重登失败数量。

### 业务接口

建议按接口族观察请求量、失败率和耗时：

- 登录：`/api/auth/wechat/login`
- 首页：`/api/users/{id}/daily-summary`
- 饮食记录：`/api/records`
- 运动记录：`/api/exercise-records`
- 体征趋势：`/api/body-metrics/trend`
- 体重导入：`/api/body-metrics/import/*`
- OCR：`/api/foods/nutrition-label/recognize`

### 外部依赖

- MySQL 连接失败和慢查询。
- 微信登录接口失败。
- 腾讯云 OCR 调用失败、超时和费用消耗。
- 云存储上传和删除失败。

## 4. 告警建议

### P0 告警

- `/readyz` 连续 3 次失败。
- 容器连续重启。
- 5xx 在 5 分钟内持续上升。
- 生产启动日志显示 `wechat.mock-enabled=true`。
- 生产启动因 `AUTH_TOKEN_SECRET` 不合规失败。

处理目标：

- 5 分钟内确认影响范围。
- 15 分钟内完成回滚或临时恢复。

### P1 告警

- OCR 失败率明显升高。
- 登录失败率明显升高。
- 数据库慢查询明显增多。
- 体重导入失败率明显升高。

处理目标：

- 30 分钟内定位是代码、配置、外部服务还是数据问题。
- 需要延期修复时，先给客服/助理同步临时口径。

## 5. 数据备份与恢复

### 备份策略

- 生产库每日自动备份。
- 重要版本发布前手动备份一次。
- 保留至少 7 天日备份和 4 周周备份。
- 内置食物和运动同步 SQL 在执行前保存执行批次和脚本版本。

### 恢复演练

每月至少演练一次：

1. 从最近备份恢复到临时库。
2. 启动后端连接临时库。
3. 调用 `/readyz`。
4. 抽样检查用户、饮食、运动、体征、日记数据。
5. 记录恢复耗时。

### 恢复目标

- RPO：最多丢失 24 小时数据。
- RTO：普通故障 2 小时内恢复服务。

如果后续用户量增长，应把 RPO 调整到 1 小时以内，并启用更频繁的 binlog 恢复策略。

## 6. 回滚策略

### 代码回滚

- 云托管保留上一个稳定镜像 tag。
- 新版本发布后 30 分钟内重点观察登录、首页、记录保存和 `/readyz`。
- 若出现 P0 故障，优先回滚镜像。

### 数据库回滚

- 表结构变更默认只前进，不直接回滚生产 DDL。
- 删除字段、删除表、批量清洗数据必须拆成多阶段：
  1. 新增兼容字段或新表。
  2. 双写或灰度读。
  3. 验证无误后停止旧字段使用。
  4. 延迟版本再清理旧结构。

### 小程序回滚

- 小程序发布前先走体验版。
- 正式版出现问题时，优先通过后端兼容处理。
- 若必须回滚小程序，保留上一版上传包和变更说明。

## 7. 常见故障处理

### 登录失败

检查顺序：

1. 小程序是否使用正确环境。
2. 云托管 header 是否包含 openid。
3. `WECHAT_APP_ID` 和 `WECHAT_APP_SECRET` 是否配置。
4. 后端日志是否有微信接口错误。
5. 生产库是否出现 `mock_openid_`。

临时处理：

- 如果是后端新版本导致，回滚镜像。
- 如果是微信接口波动，客服口径说明“登录服务短暂异常，稍后重试”。

### 首页汇总异常

检查顺序：

1. `/api/users/{id}/daily-summary` 是否 200。
2. 当日饮食和运动记录是否存在。
3. 用户目标热量是否为空。
4. 日志中是否出现食物或运动记录引用不存在。

临时处理：

- 单个用户数据异常时，先查记录引用是否被误删。
- 全量异常时，回滚镜像或恢复最近数据库备份。

### 趋势页加载慢

检查顺序：

1. 是否使用 `granularity=AUTO`。
2. `/api/body-metrics/trend` 响应耗时。
3. `body_metric_record` 索引是否存在。
4. 用户是否导入了大量历史体重。

临时处理：

- 限制前端使用 `YEAR / ALL` 的月级聚合。
- 必要时调整接口 pageSize 或增加专用聚合表。

### OCR 识别失败

检查顺序：

1. 腾讯云 OCR 密钥是否有效。
2. OCR 服务区域是否正确。
3. 图片大小和格式是否异常。
4. 当日调用量和费用是否达到限制。

临时处理：

- 识别失败不阻塞用户手动创建自定义食物。
- 客服口径强调“识别结果仅作辅助，请以包装信息为准”。

## 8. 数据维护

### 内置食物

- 正式上线后只执行 `sync-builtin-foods-latest.sql`。
- 禁止在线上执行“先删后插”的全量重建。
- 以内置食物 `source_ref` 作为稳定身份。

### 内置运动

- 正式上线后只执行 `sync-builtin-exercises-latest.sql`。
- 维护前先确认高频搜索和用户自定义运动名称。

### 云存储图片

- 健康日记删除或覆盖时会尝试清理旧图。
- 若前端清理失败，后续需要增加按 `health-diary/{userId}/{date}` 前缀巡检的后台清理任务。
- 清理任务上线前必须先 dry-run 输出待删除列表。

## 9. 运营复盘节奏

每周一次轻量复盘：

- 新增用户数。
- 首日记录完成率。
- 三日留存。
- 保存失败率。
- 搜索无结果关键词。
- 自定义食物和自定义运动新增量。
- OCR 使用量和失败率。
- 体重导入成功/失败量。

复盘输出只保留三类结论：

- 本周必须修的问题。
- 下周最值得做的产品优化。
- 暂不做的需求和原因。
