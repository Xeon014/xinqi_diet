# 体重数据导入功能设计

> 支持从其他体重管理应用导出的 CSV 文件中批量导入历史体重数据。

---

## 1. 功能概述

用户从「我的」页面进入「导入体重数据」，选择 CSV 文件后经过 **预览 → 确认 → 结果** 三步完成导入。系统自动识别文件格式，无需手动配置。

**设计目标**：覆盖市面上主流体重管理应用的 CSV 导出格式，降低用户迁移数据的门槛。

---

## 2. 用户操作流程

```
我的 → 导入体重数据 → 选择文件 → 预览解析结果 → 确认导入 → 结果展示
```

| 步骤 | 说明 |
|------|------|
| 选择文件 | 通过 `wx.chooseMessageFile` 从微信聊天记录选择 CSV/TXT 文件 |
| 预览 | 展示解析统计、逐行数据（日期、体重、状态）、重复策略选择 |
| 确认 | 发送选中行 + 重复策略到后端写入数据库 |
| 结果 | 展示导入/跳过/覆盖数量，可查看历史或继续导入 |

---

## 3. 两阶段 API 设计

### 3.1 预览接口

```
POST /api/body-metrics/import/preview
```

- 接收 `fileName` + `fileContent`（CSV 文本），通过 JSON body 传输。
- 仅解析不写入，返回每行数据的解析结果和错误信息。

### 3.2 确认接口

```
POST /api/body-metrics/import/confirm
```

- 接收已解析的 `rows`（date + weightKg）+ `duplicatePolicy`（SKIP / OVERWRITE）。
- 批量写入数据库，返回导入/跳过/覆盖计数。

**为什么采用两阶段**：预览接口让用户在导入前检查解析结果，确认接口只接收结构化数据（date + weightKg），杜绝了原始 CSV 重新解析带来的不一致风险。

---

## 4. CSV 智能解析策略

### 4.1 分隔符自动检测

统计第一行（表头）中逗号、分号、Tab 的数量，取最多的作为分隔符。

| 分隔符 | 来源应用示例 |
|--------|-------------|
| `,`（逗号） | Withings、Samsung Health、MyFitnessPal、Google Fit |
| `;`（分号） | openScale |
| `Tab` | 部分中文应用导出（如华为运动健康） |

### 4.2 列名匹配

对表头进行规范化（小写、去空格），按关键词匹配日期列和体重列。

**日期列候选**（匹配优先级从高到低）：

| 列名关键词 | 来源应用示例 |
|-----------|-------------|
| `date` | 通用 |
| `datetime` | 通用 |
| `start_time` | Google Fit |
| `时间` / `日期` / `记录时间` | 中文应用 |

**体重列候选**（匹配优先级从高到低）：

| 列名关键词 | 来源应用示例 |
|-----------|-------------|
| `weight` | Withings、Samsung Health、MyFitnessPal |
| `体重` | 华为运动健康、薄荷健康、Keep |
| `body_mass` | Google Fit |

### 4.3 单位检测

从体重列名中提取单位标识，并自动转换为 kg：

| 列名特征 | 检测单位 | 转换规则 | 来源示例 |
|---------|---------|---------|---------|
| 包含 `斤`（如 `体重 (斤)`） | 斤 | 值 ÷ 2 | 部分中文体重应用 |
| 包含 `lbs` 或 `磅` | lbs | 值 × 0.4536 | MyFitnessPal、Fitbit（美国区） |
| 包含 `kg` 或无单位标识 | kg | 保持不变 | Withings、Samsung Health |

### 4.4 日期格式自动检测

按以下顺序依次尝试解析，首次成功即采用：

| 格式 | 示例 | 来源应用 |
|------|------|---------|
| `yyyy-MM-dd` | `2025-03-31` | Withings、通用 |
| `yyyy-MM-dd'T'HH:mm:ss` | `2025-03-31T07:30:00` | Google Fit、Apple Health 导出 |
| `yyyy/MM/dd` | `2025/03/31` | 部分中文应用 |
| `yyyy/M/d` | `2025/3/1` | 部分中文应用（不补零） |
| `MM/dd/yyyy` | `03/31/2025` | Samsung Health（美国区） |
| `dd.MM.yyyy` | `31.03.2025` | openScale（欧洲格式） |

含时间部分的格式仅提取日期部分，因为本应用以天为粒度存储体重数据。

### 4.5 数据校验规则

| 规则 | 说明 |
|------|------|
| 体重范围 | 20.0 kg ~ 300.0 kg（转换后） |
| 日期限制 | 不允许未来日期 |
| 空行 | 自动跳过 |
| 无法解析的行 | 标记错误但继续解析后续行 |

---

## 5. 重复记录处理

同一日期已有体重记录时，支持两种策略：

| 策略 | 行为 | 默认 |
|------|------|------|
| `SKIP`（跳过） | 保留已有记录，跳过导入行 | **默认** |
| `OVERWRITE`（覆盖） | 用新值更新已有记录的体重和更新时间 | 用户手动选择 |

---

## 6. 支持的数据格式清单

以下列出已验证兼容的应用导出格式：

### 6.1 通用逗号分隔格式（kg）

```csv
date,weight
2025-01-01,70.5
2025-01-02,71.0
```

兼容应用：Withings、Samsung Health、通用工具导出。

### 6.2 Tab 分隔 + 斤单位

```csv
日期	体重 (斤)	体重变化 (斤)	状态	备注
2025-01-01	130.0	0	正常
2025-01-02	131.5	1.5	正常
```

兼容应用：部分中文体重管理应用。

说明：非必要列（体重变化、状态、备注）自动跳过，仅提取日期和体重。

### 6.3 分号分隔格式（openScale）

```csv
DateTime;Weight;Fat;Water;Muscle
31.03.2025;75.5;18.5;;35.2
```

兼容应用：openScale。仅提取 DateTime 和 Weight 列，Fat 等可选列忽略。

### 6.4 含时间戳的 ISO 格式

```csv
start_time,weight
2025-01-01T07:30:00,70.5
2025-01-02T08:15:00,71.0
```

兼容应用：Google Fit、Apple Health（经第三方转换工具导出 CSV）。

### 6.5 lbs 单位格式

```csv
date,Weight (lbs)
2025-01-01,155.0
2025-01-02,156.5
```

兼容应用：MyFitnessPal（美国区）、Fitbit。

### 6.6 斜杠日期格式

```csv
date,weight
2025/1/1,70.5
2025/1/2,71.0
```

兼容应用：部分中文应用。

---

## 7. 不支持的格式

| 格式 | 说明 | 建议 |
|------|------|------|
| Apple Health XML（export.xml） | 原生导出为 XML 格式 | 使用第三方转换工具（如 Health Auto Export）转为 CSV 后导入 |
| Huawei Health 原生导出 | 导出功能受限 | 尝试导出 CSV 或手动复制数据 |
| JSON 格式 | Fitbit API 等使用 JSON | 使用在线转换工具转为 CSV |
| Excel / Numbers 格式 | 二进制格式 | 先另存为 CSV 再导入 |

---

## 8. 技术实现要点

### 8.1 后端架构

遵循现有 6 层架构，核心逻辑集中在 `app/metric/WeightImportService`：

- CSV 解析使用纯 Java 实现（无第三方依赖），按整份文件扫描，支持引号字段、转义引号，以及引号字段中的换行。
- 批量插入使用 MyBatis `<foreach>` 实现，每 500 条一批。
- 重复记录查询使用 `WHERE record_date IN (...)` 单次查询，避免 N+1 问题。

### 8.2 前端实现

- 文件选择：`wx.chooseMessageFile`（从微信聊天记录选取）
- 文件读取：`wx.getFileSystemManager().readFileSync`（UTF-8）
- 数据传输：读取 CSV 文本内容后通过 JSON body 发送，复用现有 `request.js` 基础设施

### 8.3 限制

- 单次最多导入 1000 条非空数据行，超过上限时预览阶段直接拒绝
- 文件大小上限 2MB
- 仅支持体重数据（围度等其他指标暂不支持导入）
- 仅支持 CSV 纯文本格式
