# 产品体验优化路线图

> 目标：把产品设计方向拆成可排期、可验收、可回归的迭代项，避免只停留在“优化体验”的抽象描述。

## 1. 产品定位与核心闭环

`心栖轻食` 当前更适合定位为“轻量健康记录 + 趋势反馈”工具，而不是专业营养诊疗工具。产品重点应放在三个闭环：

- 低成本记录：用户能快速完成饮食、运动、体重和日记记录。
- 即时反馈：用户能在首页快速判断今天热量是否偏离目标。
- 长期趋势：用户能在趋势页看到体重、BMI 和围度变化，并回到健康档案调整目标。

当前不建议优先扩展社交、社区、复杂食谱推荐或付费咨询。这些能力会显著增加内容、审核和运营复杂度，但对当前核心记录闭环的帮助有限。

## 2. 北极星指标与分层指标

### 北极星指标

- 周活跃记录用户数：最近 7 天内至少完成 3 天记录的用户数。

### 过程指标

- 首次建档完成率：新用户进入引导页后，完成或跳过建档并进入首页的比例。
- 首日记录完成率：新用户首日完成至少 1 条饮食或体重记录的比例。
- 三日留存：首次使用后第 3 天仍有记录行为的用户比例。
- 记录效率：从首页点击添加到保存饮食/运动记录的中位耗时。
- 趋势使用率：有 3 条以上体重记录的用户中，访问趋势页的比例。

### 质量指标

- 保存失败率：饮食、运动、体重、日记保存失败次数 / 保存请求次数。
- 登录恢复失败率：401 后自动重登仍失败的比例。
- 关键页面空状态误触率：空状态页中用户无有效操作返回的比例，可先通过客服反馈和手测记录近似评估。

## 3. P0 体验改进：记录链路更快

### 首页一屏判断

- 调整首页总览卡信息层级：主数字只突出净热量，目标/剩余/超出作为次级信息。
- 当日状态标签只在有明确规则时展示，例如 `超出目标`、`记录不足`、`目标内`。
- 首页无记录时保留短文案，不增加教学式说明。

验收口径：

- 小屏机型首屏能同时看到日期、净热量、主要添加入口。
- 无目标热量用户不展示误导性“剩余”判断。
- 文案不出现编码占位或英文占位。

### 添加记录入口稳定

- 首页主添加按钮保持“添加记录”语义，展开后提供饮食、运动、体重、日记。
- 推荐餐次只作为默认值，不让按钮文案频繁随时间变化。
- 复制记录继续采用“预填后确认保存”，不直接写库。

验收口径：

- 早/午/晚/夜四个时间段进入饮食记录时，默认餐次正确。
- 复制记录不会产生未确认的新记录。
- 用户从首页到保存一条常见食物记录，目标耗时小于 20 秒。

## 4. P1 体验改进：趋势反馈更有用

### 趋势页解释策略

- `MONTH` 展示日级点，适合看短期波动。
- `YEAR / ALL` 展示月级聚合点，适合看长期方向。
- 指标卡展示最近值，不跟随图表选点；图表顶部展示选中点。

验收口径：

- `ALL` 不因历史记录增长导致首屏长时间等待。
- 年和所有时间横轴只显示首尾月份，保持可读性。
- 用户能区分“记录指标”和“查看历史明细”两个动作。

### 健康档案与趋势联动

- 健康档案页保存目标方案后，提供“查看趋势”入口。
- 趋势页新增体重后，若当天体重改变，应回写当前体重并刷新目标方案计算口径。
- BMI 只作为体重和身高派生指标，不支持手动新增。

验收口径：

- 体重记录后趋势页、首页体重卡、健康档案当前体重口径一致。
- 身高缺失时 BMI 明确展示待记录，不报错。

## 5. P2 体验改进：导入与识别降低迁移成本

### 体重导入

- 导入预览页突出三类结果：可导入、重复、错误。
- 确认导入后展示导入摘要：新增多少、跳过多少、覆盖多少。
- 对日期格式、未来日期、异常体重值给短文案解释。

验收口径：

- CSV / XLSX 均能识别日期和分钟级时间。
- 重复策略选择后结果可预测。
- 用户能在导入前知道哪些行不会被写入。

### 营养成分表识别

- 识别结果只作为自定义食物草稿，不直接保存。
- 置信度低时突出“需人工确认”，不制造确定性错觉。
- 缺失字段以“待补充：热量/蛋白质”等短文案提示。

验收口径：

- 识别失败不会阻塞手动创建自定义食物。
- 低置信度结果不会默认覆盖用户已填写字段。

## 6. P3 运营能力：从工具到可持续使用

### 用户教育

- 不在高频页面堆解释文案。
- 把复杂说明放到客服问答基线、引导页和空状态短提示。
- 只在用户遇到阻塞时展示解释，例如 OCR 低置信、导入错误、目标方案过激。

### 留存触点

- 每日记录提醒应由用户显式开启，不默认打扰。
- 提醒内容以“今天还没记录体重/晚餐”为主，不做焦虑化文案。
- 当用户连续记录 3 天后，可展示轻量正反馈，不引入勋章系统。

### 内容维护

- 内置食物和运动数据按“高频、准确、可维护”原则维护。
- 每次扩充内置库前，先看搜索无结果关键词和自定义食物高频名称。
- 不追求一次性覆盖所有食品，优先覆盖常见主食、蛋白、饮品和运动项目。

## 7. 推荐排期

### 第一期：稳定与清晰

- 首页总览卡层级调整。
- 添加记录入口语义稳定。
- 趋势页动作语义优化。
- 导入摘要补齐。

### 第二期：效率与反馈

- 食物/运动搜索结果排序优化。
- 最近记录和历史记录合并策略评估。
- 趋势页增加长期变化摘要，例如“近 3 个月下降 1.2kg”。

### 第三期：运营与增长

- 建立搜索无结果与自定义食物数据观察表。
- 增加可选记录提醒。
- 增加客服/助理常见问题回写机制。

## 8. 不建议近期投入的方向

- 社区、动态、评论等社交能力。
- 复杂食谱生成或专业诊疗建议。
- 大量营销型首页或内容流。
- 多端同步后台管理系统，除非已有明确运营人员和数据维护需求。

这些方向会增加审核、合规、运营和维护成本，但当前阶段对“记录效率”和“趋势反馈”的改善有限。
