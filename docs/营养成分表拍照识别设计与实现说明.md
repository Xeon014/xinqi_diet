# 营养成分表拍照识别设计与实现说明

> 目标：说明“营养成分表拍照识别”能力为什么这样设计、当前实现到了哪一步、后续接页面时应如何理解边界。

## 1. 背景与目标

当前饮食记录主链路仍以手动搜索和手动创建自定义食物为主。  
这对包装食品、饮料、酸奶、代餐等“用户手里已经有营养成分表”的场景不够高效。

因此本次新增能力的目标不是“自动生成一条饮食记录”，而是：

- 用户上传一张营养成分表照片
- 后端识别出名称、热量、蛋白质、脂肪、碳水等字段
- 归一化成可用于“创建自定义食物”的草稿
- 返回置信度与缺失字段，交给前端做人工确认

本能力当前只解决“识别营养成分表并生成草稿”，不解决：

- 整餐照片识别
- 条码扫码识别
- 直接自动入库
- 自动创建饮食记录

## 2. 为什么采用这条方案

本次采用的是：

- `腾讯云表格 OCR` 优先
- `腾讯云高精度通用 OCR` 兜底
- `后端规则解析与单位换算`

没有直接上多模态大模型，原因是：

- 营养成分表本身是强结构化内容，OCR + 规则更可控
- 单次识别成本更低
- 输出稳定性更适合首版落地
- 当前需求是“生成草稿”，不是“完全自动理解包装”

本方案的核心判断是：  
营养成分表识别的难点不在“看懂食品知识”，而在“把 OCR 结果可靠地归一化”。

## 3. 当前实现范围

当前仓库已落地的部分：

- 后端新增识别接口：`POST /api/foods/nutrition-label/recognize`
- 支持请求体传 `imageUrl` 或 `imageBase64`
- 新增腾讯云 OCR 适配器，支持：
  - 表格识别
  - 高精度通用 OCR 兜底
  - 开发环境 mock
- 新增规则解析器，支持：
  - 识别食品名称
  - 识别 `能量 / 蛋白质 / 脂肪 / 碳水化合物`
  - 识别基准口径：`每100克 / 每100毫升 / 每份`
  - `kJ -> kcal`
  - `每份 -> 每100g/ml` 的基础换算
  - 缺失字段与置信度输出
- 小程序服务层新增识别方法，供后续页面接入

当前还没有落地的部分：

- 拍照页面入口
- 图片压缩与上传策略的页面级封装
- 识别结果回填到自定义食物表单的完整交互
- 真实样本下的字段规则持续调优

## 4. 核心流程

### 4.1 输入约束

请求只允许两种输入方式之一：

- `imageUrl`
- `imageBase64`

二者必须且只能传一个。  
如果同时为空，或同时传值，后端直接返回参数错误。

### 4.2 识别流程

后端处理顺序如下：

1. Controller 接收识别请求并校验登录态
2. `NutritionLabelRecognitionService` 做输入归一化
3. `TencentNutritionLabelOcrClient` 先调用腾讯云表格 OCR
4. 将表格单元格整理成行文本
5. 如果表格识别结果过少，或营养关键词不足，则回退调用高精度通用 OCR
6. `NutritionLabelParser` 解析文本并生成结构化草稿
7. 返回响应对象给前端

## 5. 规则解析设计

### 5.1 字段识别

当前解析器按“关键词 + 值单位”做抽取：

- 能量：`能量`、`热量`
- 蛋白质：`蛋白质`
- 脂肪：`脂肪`
- 碳水：`碳水化合物`、`碳水`

食品名称优先从以下文案提取：

- `食品名称: xxx`
- `产品名称: xxx`

如果没有明确名称行，则尝试从非营养字段的短文本中猜测。

### 5.2 基准口径识别

当前支持三种营养表口径：

- `每100克`
- `每100毫升`
- `每份`

处理策略：

- `每100克` 直接归一化为 `quantityUnit=G`
- `每100毫升` 直接归一化为 `quantityUnit=ML`
- `每份(250ml)` 或 `每份(30g)` 会尝试换算为每 100 单位

若只识别到“每份”但没有份量，则不能安全换算，此时会：

- 保留 `baseType=PER_SERVING`
- 返回低置信度
- 将营养字段视为缺失

### 5.3 数值换算

当前支持的换算：

- 热量若为 `kJ`，统一换算为 `kcal`
- 蛋白质、脂肪、碳水若识别到 `mg`，统一换算为 `g`
- 每份口径按 `value * 100 / servingAmount` 换算为每 100 单位

### 5.4 异常值判断

当前实现了基础异常判断，用于拉低置信度：

- 单项三大营养素明显大于 `100`
- 三项营养素合计明显大于 `110`
- 热量和三大营养素折算出的理论热量偏差过大

这些规则当前只用于输出 `LOW` 置信度，不直接报错。

## 6. 响应设计

当前接口返回的是“识别草稿”，不是数据库实体。

关键字段包括：

- `foodName`
- `normalizedCaloriesPer100`
- `normalizedProteinPer100`
- `normalizedCarbsPer100`
- `normalizedFatPer100`
- `quantityUnit`
- `baseType`
- `recognizedBaseText`
- `servingAmount`
- `servingUnit`
- `confidenceLevel`
- `missingFields`
- `enginesUsed`
- `rawTextLines`

设计意图：

- 前端可以直接把归一化字段回填到自定义食物表单
- 同时保留原始文本，便于用户复核
- `missingFields` 可用于提示哪些字段还需要人工补充

## 7. 腾讯云 OCR 接入说明

当前通过腾讯云 OCR 官方签名方式直接调用接口，未引入额外 SDK。

使用能力：

- 表格识别：主链路
- 高精度通用 OCR：兜底

配置项：

- `TENCENT_OCR_SECRET_ID`
- `TENCENT_OCR_SECRET_KEY`
- `TENCENT_OCR_REGION`
- `TENCENT_OCR_MOCK_ENABLED`

环境约定：

- `dev` 默认开启 mock，便于本地开发
- `prod` 默认关闭 mock，要求真实配置密钥

## 9. 当前局限

这版实现已经能作为“后端技术打底”，但还存在明确边界：

- 未做图片预处理，斜拍、反光、折皱样本效果会波动
- 名称识别仍偏保守，很多包装不会稳定识别出食品名
- 暂未识别钠、糖、膳食纤维等扩展字段
- “每份”换算目前只覆盖最常见格式
- OCR 真实返回格式在复杂表格下仍需要用样本继续校准
- 当前缺少端到端联调回归，尤其是图片上传与大图体积控制

## 10. 后续建议

如果继续推进，建议按以下顺序：

1. 接页面拍照入口，只支持单张图
2. 将识别结果回填到“自定义食物”表单
3. 收集 50 到 100 张真实样本做规则迭代
4. 再决定是否增加图片裁剪、透视矫正、多模态兜底

不建议当前阶段直接扩成：

- 条码识别
- 整餐拍照识别
- 自动创建正式饮食记录

因为这三件事的技术不确定性和产品风险都明显高于当前方案。
